# Software Development Life Cycle

## Process

| Step | SOC2 | Description | Evidence Owner(s) | Evidence and Validations |
| :--- | :--- | :--- | :--- | :--- |
| **1** | | **Requirement Approval** | | |
| **2** | | Requirement Definition | | Creation of a Jira ticket in the appropriate project.<br>Documentation according to the standard.<br>The "Acceptance Criteria" field must contain the test criteria for the requirement. |
| **3** | C52 | Requirement Validation and Approval | Product Owner<br>Business Analyst<br>Team Lead | The requirement documented by the ticket is approved by the Product Owner, Business Analyst, or Team Lead. The status transition to *Selected for development* is restricted to the evidence owners and represents the requirement approval.<br><br>The "Approver" field must contain the name of the requirement approver.<br>The ticket history must demonstrate that the "Acceptance Criteria" field was not modified after its approval — otherwise the ticket must be formally approved again.<br>Only an approved ticket can be transitioned to the "Selected for development" status. |
| **4** | | **Code Approval** | | |
| **5** | | Feature Development | Developer | According to norms and standards.<br>The pull request title on GitHub includes the Jira ticket identifier. |
| **6** | | Automated Analyses | | Automated quality and composition analyses: Analysis tooling.<br>Automated analyses must pass successfully.<br>The code must not be modified after its analysis — otherwise the analyses are invalidated and run again. |
| **7** | C52 | Peer Code Review | Developer (CODEOWNER) | Minimum of one reviewer per pull request. Self-review is not permitted.<br>Peer review includes tracking new features, code quality, performance, and security.<br>The pull request must be approved by another developer defined in the CODEOWNER file. Self-approval is not permitted.<br>The code must not be modified after its review — otherwise the approval is invalidated and must be obtained again. |
| **8** | | Code Integration | Developer | Integration of the pull request into a protected branch (e.g., main or master). The author is authorized to merge their pull request when all approvals and validations are present.<br>The code author or a CODEOWNER integrates the code.<br>If the code is modified, the required analyses and approvals must be renewed. |
| **9** | | Change Backport *(Exceptional cycle only)* | Release Operator | Opening a pull request into a protected release branch (e.g., release/7.0). Only a Release Operator is authorized to merge the pull request when all approvals and validations are present.<br>Only a Release Operator is authorized to integrate backport code into a release branch.<br>The usual automated analyses must pass successfully. |
| **10** | | Deployment to an Internal Environment | | Automatic deployment of changes to an internal staging environment. |
| **11** | | **Change Approval** | | |
| **12** | | Application Version Creation | Release Operator | Tag of the next application version. |
| **13** | C52 | Automated Analyses | | Automated quality, composition, and security analyses: Analysis tooling. Publication of results on GitHub Advanced Security.<br>SAST workflow logs must be available for at least 15 months.<br>Analysis results must be published on the GitHub Advanced Security dashboard and must be associated with the version tag. |
| **14** | | Deployment Plan Preparation | Release Operator | In the Jira Production Release (MEP) project, create a ticket corresponding to the products to be released to production. The release plan must be completed before the production deployment.<br>The "Due Date" field must correspond to the planned production deployment date.<br>The "Assignee" field must contain the Release Operator in charge of the deployment.<br>A technical risk level must be assigned and described for each environment included in the release plan. |
| **15** | C51 | Deployment to the Acceptance Environment *(Regular cycle only)* | Release Operator | The acceptance environment automatically deploys the latest version of each service.<br>The Release Operator validates that the services have been properly deployed to acceptance and indicates it in the release plan.<br>The release plan ticket is manually transitioned to the MEA status. |
| **16** | C51<br>C52 | Manual Testing | Product Owner<br>Business Analyst<br>Team Lead | Tests are performed for all tickets representing a feature or bug in a version. The test grid corresponds to the "Acceptance Criteria" field. A "Tester" field is automatically populated when the transition to Tested is performed. The transition is restricted to the evidence owners and represents the approval of the test results.<br>In the case of an exceptional deployment (hotfix), changes will be tested in staging.<br>The "Tester" field must contain the name of the requirement tester.<br>Test criteria are defined by the "Acceptance Criteria" field (step 3).<br>The history must demonstrate that an authorized tester validated the changes before the production deployment date. |
| **17** | | Functional Release Notes Writing | Product Owner<br>Business Analyst | Confluence page describing important functional changes.<br>Discretionary assessment of high-level risk associated with the features.<br>The page clearly indicates the effective date of the described features. |
| **18** | C28 | Communication of Important Changes to Clients with a Dedicated Environment | Director, Product Development<br>Product Owner<br>Business Analyst | Important changes are communicated to clients with a dedicated environment. Communication is done through a channel previously agreed upon with the client.<br>The recipient (client).<br>A screenshot of the communication, showing: The product version, the list of changes, or a document that includes the list of important changes. |
| **19** | C52 | Production Deployment Approval | Director, Product Development | An authorized person approves the entire release plan before it is applied to production.<br>The preparatory tasks of the release plan must have been checked/completed by the person responsible for the production deployment.<br>Comment in the release plan, made by the Director, Product Development, approving the production deployment.<br>If it is a dedicated environment, a screenshot demonstrating the client's approval, including the full application version, is attached to the release plan. |
| **20** | | **Production Deployment** | | |
| **21** | C52 | Production Deployment Preparation | Release Operator | Opening and approval by a Release Operator of a pull request in the infrastructure code.<br>Pull request in flux-application, approved by a Release Operator. Self-approval is not permitted.<br>The code must not be modified after its approval — otherwise the approval is invalidated and must be obtained again. |
| **22** | C48 | Pull Request Integration by the Production Deployment Manager | Release Operator | Integration of the pull request by the production deployment manager (usually on Sunday evening).<br>No action must occur between the production deployment approval and its activation.<br>The release plan checklist must have been completed and checked by the production deployment manager. |
| **23** | | Close the Jira Release Plan Ticket | Release Operator | When the production deployment is completed, close the Jira ticket by transitioning its status to released.<br>The release plan status is transitioned to "Released". |

---

## Exceptions

1. Pull requests from bots such as **Renovate** or **Dependabot** will have a Jira ticket automatically created and
   approved. Pull requests must be approved by a human developer defined in the CODEOWNER file. The usual automated
   analyses must pass successfully.
